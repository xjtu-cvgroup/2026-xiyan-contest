# 荔枝争运战策略体检

你在为 `~/dev/sour-2026` 的对战客户端做**体系级**诊断。铁律：
**单局现象不是证据，规则数学才是真值，赢局画像不可破坏。**

历史教训（为什么需要本 skill）：迭代曾陷入逐局打补丁——每个修复对着上一局
的尸体校准（V3.5 防陷阱救了 2614 局却在 2613 局白送 90 帧；V3.10 蹲刷救了
任务荒却差点毁掉领跑画像）。每加一个条件闸门都是新的过拟合面。

真图仿真样板（验证任何估值假设的标准手法，勿凭记忆写数字）：
```python
import json, sys; sys.path.insert(0, '.')   # 在 client/ 下运行
from lychee.state import GameState
from lychee.strategy import PlannerStrategy
lines = [l for l in open('../log/replay (N).txt', encoding='utf-8') if l.strip()]
start = json.loads(lines[0])
frame = next(json.loads(l) for l in lines[1:-1] if json.loads(l).get('round') == R)
gs = GameState(2744); gs.on_start(start)
gs.on_inquire({'round': R, 'phase': frame.get('phase','NORMAL'),
               'players': frame['players'], 'nodes': frame['nodes'],
               'tasks': frame.get('tasks') or [], 'contests': [], 'events': [],
               'actionResults': [], 'weather': frame.get('weather') or {'active': [], 'forecast': []}})
st = PlannerStrategy(); print(st.planner.plan(gs), st.main_action(gs))
```

## 流程（四阶段，顺序执行）

### 一、全量盘点（证据，不是轶事）

1. `python3 tools/replay_analyzer.py log/ --all` 拿汇总表；对可疑局跑单局诊断。
2. 按对手画像分桶（竞速型/设卡型/真人），统计各分项差的**分布**而非单点。
3. 结论只承认出现在 **≥3 局或 ≥2/3 败局** 中的模式；单局异常记入观察清单但
   不得直接触发代码修改。
4. 特别标注：我们**赢了或几乎赢**的局的共同画像（当前已知：赢下 S02 起跑窗口
   → 双冰官道领跑）。这是不可破坏资产。

### 二、规则对账（任务书是唯一真值）

1. 用任务书 §7 重算分数模型，检查 `planner.py` 常量是否漂移（耗时系数、
   鲜度损耗、里程碑、时间税表、风化表——都曾手抄，可能错）。
2. 列出**未开发的规则机制**及其分值上限：悬赏(100)、情报/INTEL、破关令、
   窗口牌克制细节、T04 相邻清障、平分决胜项……体系级增量往往藏在没用过的
   机制里，而不是已有机制的第 8 个闸门里。
3. 检查对分数结构的杠杆排序是否仍然成立（送达 240 与任务 90 基础分的耦合
   是最大杠杆；用时分斜率仅 0.117/帧）。

### 三、三角验证（每个候选问题必须归到一类）

对第一阶段的每个系统性模式，用真图仿真判定属于哪类，**不许跳过仿真**：
- **代码 bug**：代码行为 ≠ 设计意图（如度量不对称、字段缺失、从未触发的分支
  ——历史上主动设卡因 ETA=0 上线数版从未运行）。
- **估值错误**：模型定价 ≠ 规则数学（如帧价值漏项、绕路公式盲区）。
- **策略空白**：规则机制未被利用（如任务刷新跟随车队身后的农场效应）。
- **运气/对手特异**：S02 窗口这类五五开，或只有某一个 demo 才有的行为
  ——**归入此类的不修**。

### 四、提案与反过拟合闸门

每条修改提案必须逐项通过，任一不过即降级为"观察项"：

| 闸门 | 要求 |
|---|---|
| 证据门 | ≥3 局出现，或纯规则数学可证（与对局无关） |
| 期望值门 | 用分数模型量化：预期 ±X 分，写出算式 |
| 回归门 | 真图仿真所有存档地图的开局决策 + `selftest.py` 全绿 + **赢局画像不变**（对着赢局回放确认关键决策点不受影响） |
| 复杂度门 | 参数调整 > 现有机制的门控 > 新机制；新增条件分支必须说明"对未见过的对手为什么也成立"。删代码优于加代码 |
| 泛化门 | 回答：这个改动帮助对抗的是**对手类型**还是**某个具体 demo 的怪癖**？后者不修 |

产出格式：按期望值排序的提案列表，每条含：问题归类 / 证据（局数+帧号）/
量化收益 / 回归验证结果 / 复杂度评级；最后附"本次拒绝的候选与拒绝理由"
（防止下次重复分析）。

## 红线

- 不许为最近一局的单一现象新增条件分支。
- 不许在未跑 selftest + 真图开局回归前给出"可以打包"结论。
- 版本水印必须递增（`client/lychee/version.py`），文档 §10/§11 同步。
- 提案里必须至少考虑一次"删掉某个现有闸门是否更好"——闸门也会过期
  （历史上奇偶让行从救命机制变成白送先手）。
